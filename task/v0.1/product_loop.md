# product_loop.md

> 产品最小闭环定义文档  
> 版本：V0.1  
> 日期：2026-07-09  
> 产品：汽车日报 Agent / 汽车市场情报报告工作台

---

## 1. 产品定位

本产品不是通用聊天机器人，而是一个面向汽车市场情报的本地报告生成工作台。

它的核心价值是：

> 将用户的自然语言汽车分析需求，稳定地转化为基于真实数据、可校验、可归档、可追问的结构化报告。

产品不追求回答所有问题，而是优先保证以下三件事：

1. 任务边界清楚；
2. 数据来源可信；
3. 报告产物可复用。

---

## 2. 产品与通用 Chatbot 的区别

| 维度 | 通用 Chatbot | 本产品 |
|---|---|---|
| 核心目标 | 回答用户问题 | 生成汽车市场情报报告 |
| 输入方式 | 任意自然语言 | 任务型自然语言 |
| 输出结果 | 对话回答 | Markdown 报告 + 报告索引 + 可追问上下文 |
| 数据依据 | 模型知识 / 临时搜索 | 本地结构化数据 + 数据校验结果 |
| 成功标准 | 回答看起来有帮助 | 报告可验证、可归档、可继续使用 |
| 失败处理 | 解释无法回答 | 明确标注意图不清、数据缺失、品牌不支持或生成失败 |

一句话定义：

> Chatbot 是“通用回答者”，本产品是“汽车情报报告流水线的自然语言入口”。

## 3. 当前最小支持范围

V0.2 阶段只稳定支持三类任务。

### 3.1 品牌日报

用户希望了解某个品牌在当前数据周期内的市场表现。

典型输入：

```text
生成比亚迪今日市场日报
看看小米今天表现怎么样
理想品牌日报
```

必要输入：

| 字段 | 是否必需 | 示例 |
|---|---|---|
| report_type | 必需 | brand |
| brand | 必需 | 比亚迪 / 小米 / 理想 |
| date | 可选 | 默认使用最新可用数据日期 |

输出：

- 品牌市场表现报告；
- 核心车型表现；
- 价格、销量、排名等关键数据；
- 风险与限制说明；
- 下一步建议。

---

### 3.2 车型对比

用户希望比较两个或多个车型。

典型输入：

```text
小米 SU7 和小米 YU7 对比
理想 L6 和问界 M7 哪个更值得关注
帮我比较蔚来 ES6 和小鹏 G6
```

必要输入：

| 字段 | 是否必需 | 示例 |
|---|---|---|
| report_type | 必需 | compare |
| series | 必需 | 小米 SU7、小米 YU7 |
| date | 可选 | 默认使用最新可用数据日期 |

输出：

- 对比表格；
- 价格区间对比；
- 销量或热度对比；
- 各自优势与短板；
- 适合人群；
- 购买或关注建议。

---

### 3.3 筛选购买建议

用户给出预算、车型、能源类型等条件，希望得到推荐。

典型输入：

```text
20 万以内新能源 SUV 推荐
15-25 万适合家用的车有哪些
预算 30 万，想买纯电 SUV
```

必要输入：

| 字段 | 是否必需 | 示例 |
|---|---|---|
| report_type | 必需 | filtered |
| budget | 强烈建议 | 20 万以内 / 15-25 万 |
| body_type | 可选 | SUV / 轿车 / MPV |
| energy_type | 可选 | 纯电 / 插混 / 增程 / 燃油 |
| usage_scenario | 可选 | 家用 / 通勤 / 商务 |

如果缺少预算，系统应优先追问，而不是直接生成宽泛报告。

输出：

- 符合条件的车型列表；
- 推荐优先级；
- 每个车型的推荐理由；
- 不确定性说明；
- 后续筛选建议。

## 4. 暂不作为主路径的能力

以下能力可以保留，但不作为 V0.2 的核心验收目标。

| 能力 | 当前策略 |
|---|---|
| 全市场日报 market | 可保留，但不优先优化 |
| 单车型深度报告 series | 可由品牌日报或对比报告延伸 |
| enriched 配置/评测数据 | 暂时降级，先不作为报告必需数据 |
| 多轮闲聊 | 不支持开放闲聊，只支持围绕报告追问 |
| 多数据源交叉验证 | 暂不实现 |
| 多用户 / 云端部署 | 暂不实现 |

## 5. 产品主流程

产品最小闭环如下：

```text
用户提出任务
  ↓
系统解析任务类型
  ↓
判断是否缺少关键信息
  ↓
判断品牌 / 车型 / 条件是否被支持
  ↓
检查本地数据是否存在
  ↓
必要时抓取和清洗数据
  ↓
构建可信报告上下文
  ↓
优先使用 LLM 生成报告
  ↓
执行报告质量校验
  ↓
写入 output/*.md
  ↓
更新 managed_reports/index.json
  ↓
前端展示报告
  ↓
用户继续追问 / 归档 / 重新生成
```

## 6. 任务解析规则

用户输入后，系统必须先产出一个结构化任务对象。

建议结构：

```json
{
  "action": "run | ask | refuse | data_missing",
  "report_type": "brand | compare | filtered | market | series | followup",
  "brand": null,
  "series": [],
  "budget": null,
  "body_type": null,
  "energy_type": null,
  "date": "latest",
  "missing_slots": [],
  "reason": ""
}
```

### 6.1 action 定义

| action | 含义 | 前端表现 |
|---|---|---|
| run | 信息足够，可以执行 | 进入报告生成流程 |
| ask | 缺少关键信息 | 向用户追问一个明确问题 |
| refuse | 请求超出产品边界或不安全 | 说明无法处理 |
| data_missing | 任务合理，但数据不支持 | 说明缺失原因和可选替代方案 |

### 6.2 不允许的行为

系统不应在以下情况下直接生成报告：

1. 品牌不在支持列表中；
2. 品牌存在但 brand_id 缺失；
3. 车型无法在当前数据中匹配；
4. 用户需求缺少关键条件，例如购买建议没有预算；
5. LLM 不可用但系统未提示降级。

---

## 7. 数据可用性判断

在生成任何报告前，系统必须判断数据状态。

建议输出结构：

```json
{
  "data_status": "available | partial | missing | unsupported",
  "reason": "",
  "available_fields": [],
  "missing_fields": [],
  "data_date": "YYYY-MM-DD",
  "can_generate": true
}
```

### 7.1 数据状态定义

| data_status | 含义 | 是否允许生成 |
|---|---|---|
| available | 所需基础数据完整 | 允许 |
| partial | 可生成基础报告，但部分补充数据缺失 | 允许，但必须标注限制 |
| missing | 关键数据缺失 | 不允许，除非用户明确接受降级 |
| unsupported | 当前品牌、车型或任务不支持 | 不允许 |

### 7.2 品牌数据判断规则

| 情况 | 系统动作 |
|---|---|
| 品牌在 brands.json 且 brand_id 有效 | 正常检查数据 |
| 品牌在 brands.json 但 brand_id 为 null | data_missing |
| 品牌不在 brands.json | unsupported |
| 品牌存在但当日数据缺失 | 尝试抓取；抓取失败则 data_missing |

### 7.3 enriched 数据处理规则

在 V0.2 阶段，enriched 数据不是报告生成的必要条件。

处理规则：

1. enriched 匹配成功：可以加入配置或评测补充小节；
2. enriched 匹配失败：不在报告正文中反复暴露失败提示；
3. enriched 状态只记录在 metadata 或调试 trace 中；
4. 报告正文只需说明“本报告基于销量、价格、排名数据生成”。

## 8. 报告生成标准

每份正式报告必须满足最低质量标准。

### 8.1 通用结构

```markdown
# 报告标题

## 核心摘要

## 关键数据

## 主要发现

## 分析判断

## 风险与限制

## 下一步建议
```

不同报告类型可以增加额外章节，但不能少于以上核心结构。

---

### 8.2 最低内容要求

| 要求 | 标准 |
|---|---|
| 标题 | 必须包含报告对象和报告类型 |
| 核心摘要 | 至少 3 条要点 |
| 数据表 | 至少 1 张结构化表格 |
| 分析判断 | 至少 3 条，不得只复述数据 |
| 风险与限制 | 必须说明数据缺口或口径限制 |
| 下一步建议 | 至少 2 条可执行建议 |
| 字数 | 默认不少于 600 字，除非用户要求简短 |

---

### 8.3 生成模式

系统应优先使用 LLM 生成报告，规则模板只作为兜底。

建议 metadata：

```json
{
  "generation_mode": "llm | llm_retry | rule_fallback",
  "llm_used": true,
  "fallback_reason": null,
  "quality_score": null
}
```

### 8.4 LLM 失败处理

流程：

```text
第一次 LLM 生成失败
  ↓
重试一次
  ↓
仍失败
  ↓
使用规则模板兜底
  ↓
在前端和 metadata 中明确标注降级
```

前端提示示例：

```text
LLM 当前不可用，已生成基础数据报告。
报告包含销量、价格和排名信息，但不包含完整深度分析。
```

## 9. 报告索引规则

每次生成报告后，必须写入 managed_reports/index.json。

建议字段：

```json
{
  "id": "",
  "title": "",
  "report_type": "brand | compare | filtered | market | series | followup",
  "entities": {
    "brand": null,
    "series": [],
    "budget": null,
    "body_type": null,
    "energy_type": null
  },
  "file_path": "",
  "data_date": "YYYY-MM-DD",
  "created_at": "",
  "status": "active | archived",
  "data_status": "available | partial | missing | unsupported",
  "generation_mode": "llm | llm_retry | rule_fallback",
  "llm_used": true,
  "parent_report_id": null,
  "followup_type": null
}
```

### 9.1 标题提取规则

标题必须来自 Markdown 第一个一级标题 `#`。

不允许把以下内容作为报告标题：

```text
核心摘要
销量前十
车型表现
主要发现
同品牌参考
风险与限制
下一步建议
```

如果报告正文没有一级标题，系统应根据任务对象自动生成标题。

示例：

```text
比亚迪品牌市场日报｜2026-07-09
小米 SU7 vs 小米 YU7 车型对比报告｜2026-07-09
20 万以内新能源 SUV 购买建议｜2026-07-09
```

## 10. Follow-up 规则

follow-up 不是开放聊天，而是围绕已有报告继续生成补充报告。

### 10.1 follow-up 上下文结构

不允许把上一份完整报告递归塞入新上下文。

建议只保存轻量上下文：

```json
{
  "source_report_id": "",
  "source_report_title": "",
  "source_report_type": "brand | compare | filtered | market | series",
  "source_entities": {
    "brand": null,
    "series": [],
    "budget": null,
    "body_type": null,
    "energy_type": null
  },
  "source_summary": "150 字以内摘要",
  "user_followup": "用户本次追问内容",
  "created_at": ""
}
```

### 10.2 follow-up 类型

| followup_type | 用户意图 | 系统动作 |
|---|---|---|
| expand | 展开解释原报告某部分 | 基于原报告续写 |
| compare | 引入新对象比较 | 进入 compare 任务 |
| refilter | 改变筛选条件 | 进入 filtered 任务 |
| regenerate | 要求使用新数据重跑 | 重新检查数据并生成 |
| clarify | 用户追问不明确 | 追问用户 |

### 10.3 follow-up 文件规则

follow-up 必须生成新文件，不覆盖原报告。

metadata 中必须记录：

```json
{
  "parent_report_id": "",
  "followup_type": "expand | compare | refilter | regenerate | clarify"
}
```

## 11. 前端最小体验规则

前端不需要复杂，但必须让用户看清楚系统正在做什么。

### 11.1 输入区提示

首页输入框附近展示当前支持能力：

```text
你可以让我生成：
1. 品牌日报：如“生成比亚迪今日市场日报”
2. 车型对比：如“小米 SU7 和小米 YU7 对比”
3. 筛选建议：如“20 万以内新能源 SUV 推荐”
```

### 11.2 任务状态展示

任务执行时至少展示：

```text
正在解析任务
正在检查数据
正在生成报告
正在校验报告
报告已生成
```

### 11.3 报告列表展示字段

报告列表至少展示：

| 字段 | 示例 |
|---|---|
| 标题 | 比亚迪品牌市场日报 |
| 类型 | brand |
| 对象 | 比亚迪 |
| 数据日期 | 2026-07-09 |
| 数据状态 | available / partial |
| 生成模式 | LLM / 模板兜底 |
| 状态 | active / archived |

## 12. 失败处理规则

系统失败时必须给出明确原因和下一步选择。

### 12.1 意图不清

```text
我还不能确定你想生成哪类报告。
你是想看品牌日报、车型对比，还是按预算筛选购买建议？
```

### 12.2 缺少预算

```text
要生成购买建议，需要先知道你的预算范围。
例如：20 万以内、15-25 万、30 万左右。
```

### 12.3 品牌未接入

```text
当前品牌暂未接入数据源，无法生成可信报告。
你可以查看已支持品牌，或改为生成整体市场报告。
```

### 12.4 brand_id 缺失

```text
该品牌已在品牌列表中，但缺少 brand_id，暂时无法抓取数据。
请先补全 brand_id 后再生成报告。
```

### 12.5 LLM 不可用

```text
LLM 当前不可用，系统将生成基础数据报告。
该报告只包含结构化数据整理，不包含完整深度分析。
```

### 12.6 数据抓取失败

```text
当前数据抓取失败。
你可以使用最近一次可用数据生成报告，或稍后重新抓取。
```

## 13. V0.2 验收标准

V0.2 的目标不是扩功能，而是让最小闭环稳定可信。

### 13.1 任务解析验收

使用以下输入测试：

```text
生成比亚迪日报
看一下小米 SU7
小米 SU7 和小米 YU7 对比
20万以内 SUV 推荐
特斯拉今天怎么样
问界日报
保时捷销量如何
新能源车整体表现
帮我分析一下小米
我要买车，20万左右
```

每条输入都必须稳定落入以下状态之一：

```text
run / ask / refuse / data_missing
```

---

### 13.2 数据可用性验收

| 请求 | 期望结果 |
|---|---|
| 比亚迪日报 | 正常生成 |
| 小米日报 | 正常生成 |
| 问界日报 | 提示 brand_id 缺失 |
| 保时捷日报 | 提示 brand_id 缺失 |
| 特斯拉日报 | 提示品牌未接入 |
| 不存在的品牌 | 提示不支持 |

---

### 13.3 报告质量验收

随机生成 10 份报告，目标如下：

| 指标 | 目标 |
|---|---|
| LLM 使用率 | ≥ 80% |
| 规则模板兜底率 | ≤ 20% |
| 空报告 | 0 |
| 少于 300 字报告 | 0 |
| 无一级标题报告 | 0 |
| 报告索引标题错误 | 0 |

---

### 13.4 follow-up 验收

基于同一份报告测试：

```text
展开讲讲销量变化
这个品牌适合买入吗
和小米对比一下
换成 20 万以内 SUV 视角
用今天数据重新生成
```

目标：

1. 不出现上下文递归嵌套；
2. 不出现用户输入乱码；
3. 每次 follow-up 生成新文件；
4. 新文件 metadata 正确记录 parent_report_id；
5. followup_type 分类正确。

## 14. 开发优先级

建议按以下顺序开发。

### P0：必须先完成

1. 明确支持三类核心任务；
2. 修正任务解析输出结构；
3. 增加数据可用性判断；
4. 品牌不支持或 brand_id 缺失时禁止生成伪报告；
5. 修复报告标题提取；
6. LLM 失败时明确提示降级。

### P1：完成后产品体验明显提升

1. LLM 生成成为主路径；
2. 规则模板只作为兜底；
3. 报告索引增加 data_status 和 generation_mode；
4. 前端报告列表展示更多关键信息；
5. follow-up 使用轻量上下文，避免递归嵌套。

### P2：后续增强

1. enriched 车型 alias 映射；
2. 5 个高频车型 enriched 匹配成功；
3. 更多品牌 brand_id 补全；
4. 新增单车型深度报告；
5. 增加测试框架。

---

## 15. 一句话版本目标

V0.2 的目标是：

> 让产品从“一个能跑起来的汽车日报 Agent”，变成“一个有清晰边界、可信数据、稳定报告产出的汽车情报工作台”。
